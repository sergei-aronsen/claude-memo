<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Actions runner to repo | CI workflow executes on GitHub infrastructure with access to repo contents |
| External actions to workflow | Third-party actions (checkout, setup-python) execute within the CI job |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Tampering | .github/workflows/ci.yml | mitigate | Pin all GitHub Actions to full SHA (not tags) per D-06 — tag-based refs can be moved to point at malicious code |
| T-04-02 | Elevation of Privilege | ci.yml permissions | mitigate | Set `permissions: contents: read` at top level — no write access to repo, no id-token, no packages |
| T-04-03 | Tampering | pull_request trigger | mitigate | Use `pull_request` (NOT `pull_request_target`) — pull_request_target runs in base branch context and can expose secrets to fork PRs |
| T-04-04 | Information Disclosure | ci.yml secrets | accept | No secrets referenced in workflow — public repo with public dependencies only, no API keys needed for lint/type-check/scan |
| T-04-05 | Spoofing | Supply chain (pip install) | accept | Dependencies installed from PyPI via pinned requirements.txt; bandit pinned to ==1.9.4; acceptable risk for CI-only tool |
</threat_model>
